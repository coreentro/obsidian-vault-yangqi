<title>【理论篇】面向小白的 Claude Skills 保姆级教程</title>

你好，我是大圣



最近Agent Skills这个概念特别火，有Anthropic 本身的 Claude Skills，Coze也跟着出了Skills



但我发现很多人看完还是懵的：这个概念到底解决什么问题，跟提示词有什么关系呢？



所以这篇文章，我想用大白话把Skills这件事讲清楚



不讲具体的技术细节，但会讲清楚它的结构和规范，c



如果你之前看过我写的提示词和MCP的文章，这篇刚好可以帮你把这些概念串起来



如果没看过也没关系，我们从头开始



# 一、先建立一个认知框架



在讲Skills之前，我想先分享一个我自己的智能体极简思考模型



AI圈的名词层出不穷：提示词、上下文工程、MCP、Skills……每隔几个月就有新概念出来。



很多人会焦虑：我是不是又落后了？根本学不完。但我想告诉你：**这些东西的本质，从来没变过**



## 1）智能体到底是什么？



我们人类对于智能体有一个最朴素的情感期待：**下达一个任务，让AI帮我们自动完成**



所以人类发明的智能体概念本质上就是一个任务解决方式：



```Plain Text
任务 → 规划（第一步做什么，第二步做什么...） → 执行（调用工具 + 理解上下文）
```



就跟人类干活一样：大脑负责规划，然后按规划去协调各种工具完成任务



AI大模型，充当的是那个大脑的角色



## 2）所有AI概念，都在解决三个问题



理解了这个框架之后，你会发现一件有意思的事：**AI圈层出不穷的新概念，本质都在解决三个问题**



1. 如何把任务规划得更精准
2. 如何调用工具更准确、更规范
3. 如何更好地理解和协同上下文

所以每次新的AI名词出现，本质都在解决上面三个问题；我们用上面这三个问题来看下面的技术



<callout emoji="✨">
1、**大模型的能力增强**：要么提升了上下文理解的能力，要么提升了任务规划的能力
2、**提示词**：既可以帮大模型做任务规划，也可以为大模型提供上下文
3、**MCP**：制定了大模型调用外部工具的规范，让大模型可以更加准确的调用外部工具
而调用外部工具也会有两种情况：分别是执行任务和获取上下文
比如生成 Word文档就是执行任务，而调用搜索引擎就是获取更多上下文
</callout>



而今天要讲的**Skills**，则是一种规范，这种规范可以把你的业务SOP沉淀下来，方便大模型更好地调用



后面会跟大家详细讲到，Skills 是一些文件夹，里面放着指令、脚本和资源，而这些指令、脚本和资源，本质就是提示词、上下文和工具调用的载体



解决的问题，仍然是**流程规划**、**工具调用**、**上下文获取**这三件事



## 3）这些概念不是对立的



每当有新的技术出来，经常有这样的声音：提示词已死，或者MCP不行了等等



这存在严重的问题，这些概念都是相互补充、相互嵌套的，而不是非此即彼的关系



- Skills里面一定会包含提示词
- Skills里面的工具调用可以遵循 MCP规范
- 提示词可以引导大模型去调用Skills
- ……

理解了这个上层逻辑，你就不会被新名词带着跑了。未来再出现什么新概念，直接往这个框架上套就行



![](https://feishu.cn/file/KWNkb1Qd8oQAnFxInm6cYcEsn95)



有了这个认知框架，我们来聊今天的主角：Skills



# 二、为什么需要 Claude Skills



##  1）Skills就是你的工作流



如果要用一个大家熟悉的概念来类比Skills，那就是**工作流**，或者叫**业务流程**



做任何一件复杂的事情，都需要固定的步骤



比如你写一篇公众号文章：先选题，再收集素材，然后列大纲，接着写初稿，最后润色排版



这些就是你的工作流



**Skills，本质上就是把这些工作流写成AI能理解的格式，让AI按照你的流程自动去执行**



## 2）为什么现在才火起来



你可能会想：工作流这个概念又不新，为什么Skills这个规范现在才出来，而且一出来就这么火？



**因为模型能力到了**



Skills是一套规范，是一本给到 AI大模型的操作说明书



在以前，模型的能力不足，让他理解一整套工作流程是有困难的



后来 Arthropic公司的 Claude模型迭代到 Claude Opus 4.5，模型的理解能力和规划能力都上了一个台阶



你给它一套规范，它真的能理解，而且能判断什么时候该调用、怎么调用



**所以Skills能火起来，不是因为这个概念有多新，而是执行这个概念的大脑终于够用了**



## 3）为什么不直接用提示词



既然Skills就是业务 SOP的封装，那我把工作流写成提示词不就行了？



首先后面会讲到，最简单的 Skill就可以由一个提示词构成



但是提示词有很典型的问题：



随着你工作流越来越复杂，你的提示词就需要背景信息、工作流程、注意事项、输出格式、参考案例等等



提示词会变得越来越长，这时候就会出现一个问题：由于模型的上下文理解能力是有限制的



如果你一次性输入的提示词过长，AI的回复速度越慢，同时也可能导致质量下降



后面会讲 Skill的设计是怎么解决这个问题的



## 4）Skills是什么？一句话总结



<callout emoji="✨">
***Skills是一种规范，让你把工作流程打包成一个技能包，AI会在需要的时候自动调用***
</callout>



这里面有两个关键点：



1. **打包成技能包**：你可以把提示词、参考文档、示例、代码脚本都放在一起，形成一个完整的技能
2. **需要时自动调用**：AI不会一股脑把所有内容都读进来，而是按需加载



接下来我们来具体拆解下 Skills的结构



# 三、Claude Skills长什么样？拆解他的结构



<callout emoji="✨">
在了解 Claude Skill之前，你必须看懂这段话：
**Claude Skill，本质上是一套规范，一套描述语言**
什么规范？**它定义了一套目录结构、一套文档格式**。你按照这个结构去写，AI就能读懂
**读懂什么？读懂你的Workflow，读懂你的工作流程**
**这就是你给 AI大模型写的操作手册**
所以大家听到Claude Skill，不要以为这是Claude专属的东西。不是的
它是Anthropic这家公司发明的一个约定、一个规范：但它已经要成为**行业标准**了
现在很多工具都在适配这套规范：
- Claude Code（Anthropic官方的）
- Cherry Studio（我们今天讲的）
- 还有Coze的Skills，底层逻辑也是一样的
**大家都在往这个规范上靠。**所以你学会了这套思路，换个工具，换个平台，逻辑是通的
</callout>



官方很明确地提出：Skills 是一些文件夹，里面放着指令、脚本和资源



那我们学习 Claude Skill的核心就是学习文件夹的结构，哪些文档是必须的，哪些文件夹是必须的，以及他们各自承担的作用是什么



## 1）Skill的文件夹目录框架



一个Skill就是一个文件夹，文件夹里面放着跟这个业务流程相关的所有东西：说明书、参考资料、示例、脚本



```Plain Text
skill-name/
├── SKILL.md           ← 入口文件（必需）
└── 打包资源（可选）
    ├── scripts/       ← 可执行脚本
    ├── references/    ← 参考文档
    └── assets/        ← 输出资源（模板、图片等）
```

## 2）SKILL.md：技能的大脑



![](https://feishu.cn/file/UOyZb39vqonp8RxgHrNcVXGOn1g)



先说 SKILL.md，**这是唯一必需的文件，也是整个技能的大脑**



AI拿到一个Skill，第一个读的就是这个文件，它由两部分组成



**第一部分：头部信息（Frontmatter）**



```YAML
name: skill-creator
description: 创建高效技能（Skill）的指南。
当用户想要创建一个新技能（或更新现有技能），
通过专业知识、工作流或工具集成来扩展 Claude 的能力时，应使用此技能
```



这里有两个字段：

- **name**：技能的名字，同时也是技能的唯一标识
- **description**：技能的描述，这个非常关键



**为什么description很关键？**



因为AI就是靠这段话来判断要不要调用这个技能的



它会把你说的话，和所有Skill的description进行匹配。如果匹配上了，它就会调用这个Skill



description是最关键的部分，AI 通过这段话来判断什么时候该激活这个技能



所以description写得越清楚、越具体，AI就越知道什么时候该用它。



官方给了一个好的description示例，它不只是说这是一个文档处理技能，而是把具体能做什么、什么时候该用，都写清楚了



```YAML
全面的文档创建、编辑和分析，支持修订模式、批注、格式保留和文本提取。
当需要处理专业文档（.docx文件）以进行以下操作时使用：
(1) 创建新文档，
(2) 修改或编辑内容，
(3) 处理修订，
(4) 添加批注，
或任何其他文档任务。
```



**第二部分：正文内容（Body）**



头部信息下面，就是具体的指令了，其实就是提示词，我举个简单示例



```Markdown
---
name: 文章审核助手
description: 当用户需要对文章进行深度分析、内容审核或质量评估时使用此技能
---

# 文章审核指南

## 你的任务
对用户提供的文章进行深度分析和审核。

## 工作流程
1. 先通读全文，理解文章主旨
2. 检查逻辑是否自洽
3. 检查是否有事实性错误
4. 给出修改建议

## 输出格式
请按以下结构输出：
- 总体评价
- 具体问题
- 修改建议

## 参考资料
- 详细的审核标准见 references/standards.md
- 常见问题案例见 references/examples.md
```



你会发现，**SKILL.md的正文，本质上就是一个结构化的提示词**

## 3）资源文件



### **scripts/：脚本目录**



这个文件夹里放的是可执行的脚本，也就是可执行代码（如 Python、Shell 或 Node.js 脚本）



**什么时候需要用脚本？**



当有些事情你教AI十遍它也可能出错的时候。比如旋转PDF的第3页。与其用文字描述怎么操作，不如直接写一个`rotate_pdf.py`脚本



脚本的好处是：



- **确定性强**：逻辑是固定的，不会出现理解偏差
- **省Token**：AI执行脚本只需要看输出结果，不需要把代码读进上下文



sccripts使得AI模型不再局限于文本生成，而是能够执行可靠的逻辑计算、文件操作或 API 调用，显著提升复杂任务的精确度



### references/：参考文档目录



这个文件夹里放的是各种参考资料，比如：



- 完整的API文档
- 数据库的表结构
- 公司的政策规范
- 详细的工作流指南



**为什么要单独放一个文件夹，而不是都写在SKILL.md里？**



因为这些内容往往很长，如果都塞进SKILL.md，每次调用都会占用大量上下文



放到references/里，AI只在需要的时候才去读



**官方有个建议：** **如果某个参考文件超过1万字，最好在SKILL.md里告诉AI可以用什么关键词去搜索，而不是让它通读全文**



### assets/：输出资源目录



这个文件夹放的是AI在输出时会用到的文件，比如：



- 公司logo图片
- PPT模板
- 前端项目的样板代码
- 字体文件



**它和references/的区别是什么？**



- references/：**是给AI读的，用来理解怎么做**
- assets/：**是给AI用的，会出现在最终输出里**



比如你让AI帮你做一份PPT，assets/里的模板文件会被直接复制和修改，而不是被阅读理解

# 四、渐进式加载：Skills最核心的设计



前面我们说过，Skills和普通提示词最大的区别是按需加载。



那AI到底是怎么按需的？它怎么知道什么时候该读哪个文件？



这就是Skills最核心的设计：**渐进式加载（Progressive Disclosure）**



## 1）三层加载机制



你可以把这个过程想象成查字典：先翻目录找到大概在哪一页，再翻到那一页看具体解释，如果解释里提到详见附录，你才会去翻附录。不会一上来就把整本字典读一遍



### 第一层：发现



AI启动时，会扫描所有可用的Skills（**Skills如何存放后面会讲，当前不用关心**）



但它不会把每个Skill的完整内容都读进来，那样太浪费上下文了，**它只读每个Skill的简历：也就是name和description。**



```YAML
---
name: 文章审核助手
description: 当用户需要对文章进行深度分析、内容审核或质量评估时使用此技能
---
```



就这么几十个字，大概100个token左右



这一步的目的是：让AI知道有哪些技能可以用，但不占用太多上下文。



### 第二层：激活



当你说了一句话，AI会拿这句话去匹配所有Skill的description。



如果匹配上了，它才会加载这个Skill的完整内容，也就是SKILL.md的正文部分



比如你说帮我审核一下这篇文章，AI发现这句话跟**文章审核助手**的description匹配，于是它就会把这个SKILL.md完整读进来。



读进来之后，AI就知道了：



- 具体该怎么做（工作流程）
- 有哪些参考资料可以用（references/里有什么）
- 有哪些脚本可以执行（scripts/里有什么）

**注意：这时候它只是知道有这些资源，但还没有真正去读**



### 第三层：按需读取



AI模型开始执行任务了



执行过程中，它会根据需要，决定要不要去读references/里的文件，要不要执行scripts/里的脚本



比如SKILL.md里写了：



```Markdown
## 参考资料
- 详细的审核标准见 references/standards.md
- 常见问题案例见 references/examples.md
```



如果AI在审核过程中觉得需要看审核标准，它就会去读`references/standards.md`



如果它觉得当前任务不需要，这个文件就不会被读取，**不需要的文件，始终不会被读取**



**具体可以看视频演示**

## 2）为什么要这样设计？



两个原因：



**1、省Token**



Token是 AI大模型最宝贵的资源，有金钱成本



如果每次调用都把所有内容都加载进来，消耗的金钱是指数级提升的



**分层加载，用多少读多少，成本最低**



**2、AI更聚焦**



AI模型的上下文窗口是有限制的，如果一次性读太多内容，AI就会选择性失忆，从而影响任务的输出质量



渐进式加载的设计，确保AI在每个阶段只看到它需要的信息，不会被无关内容干扰



## 3）两个核心原则



理解了渐进式加载，在设计 Skill的时候就会有两个原则



### 原则一：必须在SKILL.md里引用，AI才能找到



AI不会自动扫描你的Skill文件夹



你在references/里放了一个`standards.md`，如果SKILL.md里没有提到它，AI根本不知道有这个文件存在



**它只是静静地躺在那里，永远不会被用到**



所以，你必须在SKILL.md里明确写出来：



```Markdown
详细的审核标准见 references/standards.md
```



AI看到这句话，才知道：哦，原来还有这么一个文件，需要的时候可以去读



### 原则二：引用保持一层深度，不要套娃



SKILL.md → references/standards.md，这样没问题。



但如果standards.md里又引用了另一个文件，那个文件里又引用了下一个文件……



这种层层嵌套的引用，AI可能会读不完整



**保持简单：SKILL.md作为中心，直接指向所有资源，不要套娃**



<callout emoji="✨">
**官方原话是：保持引用距离SKILL.md只有一层深度**
</callout>



## 4）小结



| 加载层级 | 读取内容 | 触发条件 |
|-|-|-|
| 第一层：发现 | name + description | AI启动时自动扫描 |
| 第二层：激活 | SKILL.md完整正文 | 用户请求匹配description |
| 第三层：按需 | references/scripts/assets | 执行过程中按需读取 |



**核心理念：AI足够聪明，知道什么时候需要什么。**



**你要做的，是把东西放对位置，然后在SKILL.md里告诉它去哪找**

# 五、Skills放在哪里？存放规范



既然Skills是一个文件夹，那这个文件夹应该放在电脑的哪个位置呢？



这个问题很重要，**因为AI模型要调用Skill，首先得能找到它，**你随便放在桌面上某个角落，AI是不知道的



Skills的存放位置是有规范的，分两种：



1. **项目级**：放在当前项目的根目录下。这种 Skills 只在该特定项目内可用
2. **全局级**：放在用户根目录下的特定路径（例如 `~/.claude/skills/`）这种 Skills 对所有项目都生效，是用户个人的通用技能库



## 1）项目级存放



什么是项目级？就是把Skills放在你当前工作的项目文件夹里。



**结构是这样的：**



```Plain Text
your-project/              ← 你的项目文件夹
├── 你的项目文件...          ← 项目本身的文件，随便放
├── .claude/               ← 固定名称，必须叫这个
│   └── skills/            ← 固定名称，必须叫这个
│       ├── skill-a/       ← 你的第一个Skill
│       └── skill-b/       ← 你的第二个Skill
```



假设你有一个项目叫**我的公众号**，里面放着你写的文章、素材等



现在你想给这个项目加两个Skill：一个用来审核文章，一个用来生成配图



结构应该是这样的：



```Plain Text
我的公众号/
├── 文章/
│   ├── 第一篇.md
│   └── 第二篇.md
├── 素材/
│   └── 参考资料.pdf
├── .claude/
│   └── skills/
│       ├── article-review/
│       │   └── SKILL.md
│       └── image-generator/
│           ├── SKILL.md
│           └── assets/
│               └── style-template.png
```



**几个关键点：**



1. `.claude`是一个隐藏文件夹（前面有个点），这是规定的名称，不能改
2. `.claude`下面必须有一个`skills`文件夹，名称也不能改
3. `skills`文件夹里面，每个子文件夹就是一个Skill
4. 你的项目文件该怎么放还怎么放，跟Skills互不干扰



## 2）全局级存放



还有一种方式是全局级，就是把Skills放在用户根目录下。



```Plain Text
~/.claude/skills/
├── skill-a/
└── skill-b/
```



这种方式的好处是：不管你在哪个项目下工作，这些Skills都能被调用



适合放一些通用的、跨项目都会用到的技能



## 3）项目级还是全局级？



目前我还在探索最佳实践，但是基于隔离原则，我会推荐大家基于项目级进行 Skill存放



因为不同项目的工作流往往不一样。你写公众号的流程，和你做数据分析的流程，肯定是不同的。



项目级存放的好处是：**每个项目可以有自己专属的Skills，互不干扰**



**这本质其实就是你的文件夹存放习惯，怎样让你的项目文件存放更合理**



## 4）AI是怎么找到Skill的



你可能会好奇：我按这个规范放好了，AI怎么知道去哪里找？



答案很简单：**只要你按规范放，AI会自动扫描**



<callout emoji="✨">
在下一篇教程里我们会讲，具体该使用什么平台进行 Skill的调用
这里我们先聚焦 Skill的逻辑
</callout>



当你在**某个项目目录下**启动AI（比如用Claude Code或者其他支持Skills的工具），当你提出你的任务时，它会自动去找`.claude/skills/`这个路径



找到之后，就会扫描里面所有Skills的name和description，这就是三层加载机制的第一层：发现



所以你不需要手动告诉AI我的Skill在哪里，只要放对位置就行了



**PS：当然，我们也可以在调用工具中手动指定调用的 Skill，这会在其他篇教程详细讲解**

## 1）再看一遍那三个问题



开头我们说过，AI圈所有的概念，本质都在解决三个问题：



1. 如何把任务规划得更精准
2. 如何调用工具更准确、更规范
3. 如何更好地理解和协同上下文

那Skills解决的是哪个问题？**答案是：三个都有**



- SKILL.md里的工作流程 → 帮助任务规划
- scripts/里的脚本 → 规范工具调用
- references/里的参考文档 → 提供上下文
- 渐进式加载的设计 → 优化上下文管理

这也是 Skill强大的地方，它不是解决某一个点的问题，而是提供了一个**完整的框架**，把任务规划、工具调用、上下文管理都打包在一起了



其实也非常好理解，Skills的目的是解决一个完整的业务流程，那它就必须包含所有的技术



## 2）Skills和其他概念的关系



现在你应该能理解，为什么我在开头说这些概念不是对立的了



| 概念 | 做什么 | 和Skills的关系 |
|-|-|-|
| **提示词** | 告诉AI做什么、怎么做 | Skills的SKILL.md本质就是提示词，只是被封装起来了 |
| **MCP** | 让AI能调用外部工具 | Skills的scripts/可以按照 MCP规范，两者是配合关系 |
| **上下文工程** | 优化AI的信息输入 | Skills的渐进式加载，就是一种上下文工程的实践 |



Skills不是要取代提示词，也不是要取代MCP



**它是把这些东西整合在一起，形成一个可复用、可管理的技能包**



用一个公式来表示：



```Plain Text
Skills = 提示词 + 参考资料 + 脚本 + 渐进式加载机制
```



## 3）什么时候该用Skills？



讲了这么多，你可能想问：我到底什么时候该用Skills？



**三个判断标准：**



1. **重复性**：这件事你是不是经常做？如果每次都要重新写提示词，就该考虑封装成Skill
2. **复杂性**：流程是不是有多个步骤？需要参考多个文档？简单任务直接用提示词就够了
3. **标准化**：是不是需要保持一致的输出质量？比如团队协作、品牌规范

**反过来，这些情况不需要用Skills：**



- 一次性的简单任务，直接用提示词
- 探索性的对话，不确定要做什么
- 流程还没跑通，还在摸索阶段

## 4）不要陷入优化成瘾：先跑通，再封装



最后想给大家提个醒。



我见过很多人（包括我自己），看到Skills这种东西就兴奋了



脑子里马上浮现出一个宏大蓝图：我要打造一个智能体系统，各个Skill相互联动，自动更新、自动学习、越来越懂我



然后就开始陷进去了。不断优化配置，不断调整架构，设计各种复杂的联动逻辑



**结果大概率失败。**为什么？



模型确实比以前强了，但没有你想象的那么强。它能做好单点任务，但多个任务串联起来，不确定性就会累积。一个环节出问题，后面全崩。



更重要的是：**如果你的业务流程本身还没跑通，自动化只会把问题放大**



**先跑通，再封装。先简单，再复杂。**



从一个最简单的Skill开始：只有一个SKILL.md，只解决一个具体问题，流程不超过5步。



跑通了，验证有效了，再考虑扩展。



**Skills不是用来实现宏大构想的，而是用来固化已经验证过的流程**



记住这句话，能帮你省很多弯路
