# 写在前面



你好，我是大圣



这一整章，我会用 Claude Code 作为演示工具进行教学



但我希望你从一开始就明确一件事：**你学的不是 Claude Code，你学的是智能体的底层语言**



# 一、为什么要先说这个



因为你一定会遇到这样的情况：



你打开自媒体平台，到处都有人在说不同的工具：Claude Code、Kimi Code、Cursor、Windsurf等等



每隔几个月就会冒出一个新的工具，每个工具都说自己很强



你很容易产生一种焦虑：**我学大圣讲的这个Claude Code，会不会很快就过时了？**



我的答案是：**工具一定会过时，但工具背后的思想不会**



因为不管这些工具怎么变，它们在做的事情其实都一样



# 二、智能体的极简思考模型



不管你用的是哪个工具，不管工具怎么迭代，所有的 AI 智能体都在做同样的三件事：



**任务 → 规划 → 执行 → 完成**



**第一步：规划（Planning）**



你把任务交给 AI，AI 的大脑（大模型）会先想一下：这个任务该怎么做？先做什么后做什么？



这一步靠的是大模型的能力。模型越强，规划越准



**第二步：执行（Execution）**



规划好之后，AI 开始干活。干活的过程中，它需要两样东西：



1. **工具**：搜索引擎、地图、数据库、代码执行器



它需要调用这些外部工具来完成具体的操作



1. **上下文**：它需要知道你是谁、你的项目是什么、你之前说过什么、有哪些参考资料



这些信息帮助它更准确地理解和执行任务



这就是这张图的核心：**所有你在这一章里学到的概念，都在解决规划、工具、上下文这三个问题**



![](https://feishu.cn/file/OYp9bALYRoOV8pxNnIBcUc1mndl)



# 三、那些概念，其实都在做同一件事



我们这一章会讲到 MCP、CLAUDE.md、Commands、Hooks、Skills 这些东西



名字不同，但它们解决的问题，全部都可以归到上面那三件事里



- **提示词（Prompt）**：帮助大模型更好地**规划**任务。你告诉它你要什么、不要什么、风格是什么样的



- **CLAUDE.md**：给大模型预装**上下文**。让它一开始就知道你的项目背景、你的偏好、你的规矩



- **MCP**：给大模型接上**工具**。让它能查天气、搜网页、读数据库，获取外部信息



- **Skills**：把工具 + 上下文 + 提示词**打包在一起**。一个 Skill 里面可能同时包含提示词、参考资料、甚至会调用 MCP 工具



- **Commands**：把常用的提示词存成快捷方式，本质还是在优化你给 AI 的**指令**



- **Hooks**：用程序来保证规则的执行，本质是把**上下文中的约束**变成了确定性的代码



你会发现，这些概念不是对立的，而是互补的。它们从不同的角度，解决的都是同一个问题：



**怎么让 AI 更好地理解任务、调用工具、利用上下文**



![](https://feishu.cn/file/UgeAbypproScUKxmwjjcecDFnJd)



# 四、工具会变，协议也会变，但问题不会变



2024 年 11 月，Anthropic 推出了 MCP 协议，当时整个行业都觉得这是一个革命性的东西



2025 年底，Skills 规范出来了，很多人发现 Skills 比 MCP 更火、更好用



也许半年之后又有新的协议出来，**你会发现，协议在不断迭代**



但如果你仔细看，每一次迭代解决的问题其实是相通的：



- 怎么让 AI 更方便地调用外部工具？（MCP 在做这件事）
- 怎么让 AI 更好地理解复杂任务的执行流程？（Skills 在做这件事）
- 怎么让 AI 记住你的偏好和规矩？（CLAUDE.md 在做这件事）



而且你会发现一个趋势：**协议一定会越来越简单、越来越标准化**



就像 USB 接口的演进：



最早有各种各样的接口Mini USB、Micro USB、Type-A、Type-B，现在基本统一成了 Type-C



接口在变，但让设备之间能互相连接这个需求从来没变过



所以你不需要担心：我现在学的东西会不会白学



因为你学的不是某个具体的协议格式，你学的是：



- 大模型为什么需要上下文？
- 怎么让大模型调用外部工具？
- 怎么把重复的工作流固化下来？



这些问题，两三年内不会变



# 五、工具之间可以迁移



这些概念在不同工具之间是相通的



你在 Claude Code 里学会了怎么配置 MCP，换到 Kimi Code、Cursor、Windsurf，配置方式大同小异，都是在一个 JSON 文件里声明工具



你在 Claude Code 里理解了 CLAUDE.md 的作用，换到其他工具，它们也有类似的规范文件，名字可能不一样，但做的事情一模一样



你在 Claude Code 里学会了写 Skills，这套规范已经被定义为开放标准，越来越多的工具开始支持



**底层逻辑是通的，具体按钮的位置不同而已**



# 六、带着这个心态去学



所以接下来的课程，我希望你带着这样的心态：



**第一，不要被工具名称绑架**



我用 Claude Code 做演示，是因为它目前的生态最成熟、对这些概念的支持最完善。



但你完全可以用其他工具来学习同样的东西



**第二，关注思想，而不只是操作步骤**



每节课我都会讲具体怎么操作，但更重要的是理解：为什么要这样做？这个操作背后的思想是什么？



模板化、自动触发、封装：这些思想比任何一个具体的命令都有价值



**第三，没有人能穷举所有工具**



我不可能把每个工具都讲一遍，任何课程都做不到这件事，而且也没有必要。



工具太多了，而且还在不断冒出新的。**你需要的是掌握底层逻辑，然后自己去迁移**



# 写在最后



心里建设完毕，开始学习
