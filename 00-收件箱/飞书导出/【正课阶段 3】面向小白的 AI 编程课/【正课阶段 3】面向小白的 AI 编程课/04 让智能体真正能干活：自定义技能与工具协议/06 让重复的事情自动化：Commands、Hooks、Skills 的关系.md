<title>让重复的事情自动化：Commands、Hooks、Skills 的关系</title>

你好，我是大圣



前面几节课，我们已经学会了跟 Claude Code 对话、让它记住项目规范、给它装上外部工具



到这里，你已经能用 Claude Code 干不少事了



但如果你用了一段时间，你一定会发现一个现象：**你在重复自己**



# **一、你在重复自己**



这种重复体现在三个层面：



**第一种重复：你总在说同样的话**



每次让 Claude Code 写文章，你都要把风格要求、字数限制、格式规范重新说一遍。只有主题在变，其他全是重复的



**第二种重复：你总在叮嘱同样的规矩**



你跟它说过【不要修改 published 目录下的文件】，但聊了二十轮之后，它可能就忘了。



你又得再说一遍。甚至有时候你自己也忘了叮嘱，结果它就真的改了



**第三种重复：你总在从头引导同样的流程**



你写公众号文章的流程是固定的：先选题、再搜资料、然后列大纲、写初稿、润色、起标题。



每次你都要一步一步引导 Claude Code 走完这个流程。下次写新文章，又要从头引导一遍



这不只是 Claude Code 的问题，**这是所有工作中的通病，重复消耗你的时间和精力**



# 二、一个朴素的道理



软件工程里有一个特别著名的原则，叫 DRY：Don't Repeat Yourself，不要重复自己



这个原则听起来很技术，但其实它是一个非常朴素的道理：



> **如果一件事你做了三次以上，就该想办法把它固化下来，而不是每次都从头来**



你不需要是程序员才有这个意识。生活中你早就在这么做了：



- 你每天早上的出门流程是固定的（起床、洗漱、穿衣、出门），你不会每天重新设计这个流程
- 你点外卖有常点的店和常点的菜，你不会每次都从头翻菜单
- 你设了闹钟来提醒你，而不是每天晚上心里默念明天七点要起床



**把重复的事情固化下来，这就是效率的本质**



Claude Code 为你提供了三个工具来做这件事。它们分别解决三种不同类型的重复



# **三、每个工具，解决不同的重复**



## **1）Commands：解决重复说同样的话**



你每次让 Claude Code 写文章，都要输入一大段要求。其实这些要求每次都一样，只有主题在变



Commands 的做法很直接：**把那段固定的话存成一个文件，下次只输入变化的部分就行**



存好之后，你以后写文章只需要输入：



```Bash
/write AI赋能家庭教育
```



后面那一大段风格要求、字数限制、格式规范，全部自动填上了



**这背后的思想，叫做模板化**



把一件事拆成两部分：**不变的部分**和**变化的部分**。把不变的部分固定下来，只留一个口子给变化的部分



生活中你早就在用这个思想了



假设你经常去一家吃面的饭店，非常熟悉了。你每次的要求都是：不要葱，不要香菜，多放醋，多放辣



在你和老板熟悉了之后，你可能会跟他约定一下。你说：我今天要一个臊子面，老规矩



Commands 就是 Claude Code 里的老规矩



## 2）**Hooks：解决有些规矩它总忘**



你明确告诉过 Claude Code：不要修改 published 目录下的文件



它说好的。然后聊了二十轮，你让它改个 bug，**它顺手把 published 里的文件也改了**



你很生气：我不是说了不要改吗？



但这不能怪它。**AI 的注意力是有限的，对话越长，早期的信息权重越低。** 你说的不要修改那句话，已经被二十轮对话稀释了



那怎么办？**不靠 AI 的记忆，靠程序来保证**



Hooks 就是这个思路。你写一个脚本，配一个触发条件：当 Claude Code 试图修改文件时，先检查路径，如果是 published 目录，直接拦截，不允许执行



这个检查是 100% 确定性的。不管对话进行了多少轮，不管 AI 记没记住你的话，**程序逻辑永远不会遗漏**



**这背后的思想，叫做自动触发**



什么意思？就是你把规则配好之后，它在条件满足时自动执行，完全不依赖人的记忆



生活中最典型的例子就是闹钟



你不需要每天晚上心里默念明天早上七点要起床，你设好闹钟，它到点就响。不管你记没记住，它都会响



Hooks 就是 Claude Code 里的闹钟



## 3）Skills：解决复杂流程串不起来



前面两个工具解决的是"单点"问题：一段话的重复、一条规则的遗忘



但如果你的工作流本身就很复杂呢？



比如你写公众号文章的完整流程是：



```Bash
选题 → 搜索资料 → 列大纲 → 写初稿 → 润色 → 起标题 → 质量检查
```



这不是一句提示词能搞定的，也不是一条规则能覆盖的



这是一整套流程，涉及多个步骤、多种工具、多份参考资料



每次写新文章，你都要从头引导 Claude Code 走一遍这个流程



Skills 的做法是：**把这整套流程打包成一个技能包** 



里面包含提示词、参考文档、可执行脚本；所有这个流程需要的东西，全部封装在一起



封装好之后，Claude Code 在需要的时候自动调用这个技能包，按你设定的流程一步步执行



**这背后的思想，叫做封装**



把一个复杂的东西打包起来，对外只留一个简单的入口。你不需要知道里面每一步是怎么执行的，你只需要触发它



生活中最好的例子是洗衣机的一键洗烘



你把衣服扔进去，按一个按钮，洗衣机就自动执行浸泡、洗涤、漂洗、脱水、烘干这一整套流程。



你不需要手动控制每一步，它已经被封装好了。Skills 就是 Claude Code 里的一键洗烘

# **五、你不需要全学**



最后想说一句很重要的话：**你不需要把这三个都学会**



大部分人在大部分场景下，**Commands 就完全够用了。** 你只是想少打一些字、少重复一些话，一个 Markdown 文件就能解决



什么时候需要 Hooks？



当你有一些**绝对不能违反的规则**，而且你不放心让 AI 自己记住。比如永远不要删除某个目录，这种事情必须靠程序来保证



什么时候需要 Skills？



当你有一套**成熟的、多步骤的工作流**，而且你已经手动跑通了很多遍，想把它沉淀下来复用



**而且我想跟你说一个真实的情况**



我们前面把 Commands、Hooks、Skills 讲得好像是一个整齐的阶梯：



从简单到复杂，一级一级往上走。**但技术的发展从来不是这样提前规划好的**



现实是：当你学到 Skills 之后，你可能会发现：Skills 已经能做 Commands 能做的所有事情了。



那我还需要 Commands 吗？答案是：**可能真的不需要了**



越新的技术，往往会包含前面技术的能力。



这很正常。所以不要纠结我是不是每个都要学、每个都要用。你学了 Commands 之后，有可能后面一直都用不上它，因为你发现了更顺手的方式



**但每个东西都不会白学，**学习 Commands 的过程，让你理解了模板化这个思想。这个思想本身比工具更有价值，它会伴随你使用任何工具。



**按需学，别贪多。你的需求停在哪一层，就学到哪一层**



接下来三课，我们会逐个实战：先学最简单的 Commands，再学 Hooks，最后学 Skills



每一课都是独立的，你可以根据自己的需求选择学哪个
