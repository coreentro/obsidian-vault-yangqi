# 一、写在前面



最近 OpenClaw 特别火



各种故事满天飞：498 装机年入百万、一个人指挥 6 个数字员工、企业老板装了之后业务自动化了



我既不想吹捧，也不想贬低，这篇文章想回答几个问题：



**OpenClaw 到底是什么？它为什么这么火？我们到底要学什么？以及我们该按照什么样的顺序去学？**



先抛观点：**我们要学，但我们学的不是OpenClaw这个工具的操作**



工具一定会过时。事实上已经在出了：KimiClaw、MaxClaw，还有什么ZeroClaw、EasyClaw



所以你今天花时间学 OpenClaw 的具体怎么装、怎么配，一两个月后可能就白费了



但OpenClaw 的价值恰恰在于：**通过拆解他，你能学到一种拆解任何 AI 产品的通用思维**



如果说2025年初的DeepSeek让大家开始真正接触到AI大模型，那么2026年初的OpenClaw让大家开始真正接触到智能体



我在春节这段时间，一边给学员做课程，一边自己折腾了十几个小时把 OpenClaw 装完，再用 Claude Code 帮我理解它的架构



**我发现：OpenClaw 的智能化没有什么特殊的，它真正出圈的原因是工程化**



而这个工程化的逻辑，适用于未来很长一段时间的任何 Agent 工具



所以今天借 OpenClaw 这个热门话题，我想把这层逻辑给大家讲清楚。



学懂了，下次新工具出来，你自己就能判断：值不值得学、能不能解决我的问题、边界在哪里



# 二、OpenClaw为什么火？



OpenClaw火是因为它击中了人类对于智能体最朴素的情感期待：



**我能不能有一个助理，随时在手机上指挥它干活，然后它自己就把事情做完了？**



也就是一键自动化



OpenClaw给了这个想象一个看得见摸得着的形态。你可以把它接进飞书、钉钉等聊天软件，用手机聊天就能下指令；它可以7×24小时不停运转；它还能主动推送日报、提醒你完成任务



这三件事加在一起，再配上自媒体的大力渲染，让很多人觉得：**AI时代的数字员工，终于又来了**



所以它当然会火。这种火有它真实的价值基础，我不认为这是纯粹的泡沫



但问题也恰恰出在这里：**它火的方式，正在让很多人以错误的姿势冲进去**



# 三、拆解AI产品的第一刀：智能化 VS 工程化



我有一个习惯。每次看一个 AI 产品，我都会问自己一个问题：



**这个产品里，哪些是智能化的东西，哪些是工程化的东西？**



这个问题能把"自媒体叙事"快速还原成"产品事实"



**智能化就是AI模型本身的能力**：它能不能理解你的意图，能不能做规划，能不能自主正确调用工具把任务跑完。



这一层的突破往往来自模型本身的升级：更大的参数、更好的训练数据、更新的算法



**工程化**就是怎么把模型的能力包装成产品：怎么让用户用上、怎么让它稳定运行、怎么让它可扩展。



这一层的突破，往往来自架构设计、系统设计和产品设计



现在我们用这个框架来看OpenClaw



## 1）OpenClaw的智能化



OpenClaw可以自主配置国内外的任何大模型。它可以让模型自主规划、自主调用工具、自主获取上下文



这个逻辑，跟所有 Agent 框架的逻辑一样。



Claude Code 也是这样，Coze 也是这样，未来很长一段时间的Agent 工具也都是这样



![](https://feishu.cn/file/CqHibTyKRoIBWwxFBi5cxEVan0W)



如果非要说一下OpenClaw的框架有任何特殊性，就是他使用的一个Agent框架，叫做Pi



这个框架的核心哲学是：



<callout emoji="✨">
让Agent自己写工具扩展自己，而不是预装一堆插件
Pi认为：现代大模型已经内化了对计算机系统的理解，给他四个基础工具就够了。需要什么能力，让他自己写代码来扩展自己
而这四个基础工具就是Read、Write、Edit、Bash
所以派主动拒绝了 MCP，不是做不到，是觉得没必要。与其下载别人写的插件，不如让 Agent 自己造一个更合适的
</callout>



这个框架并没有什么特殊的，它并不是在智能化上有了什么本质的突破，只是有自己的一套哲学



它的核心仍然依赖大模型能力的突破，如果大模型的能力没有达到某个边界，这个框架意义不大



## 2）真正让OpenClaw出圈的，是三个工程化的能力



**第一个：接入了聊天渠道**



OpenClaw 可以接入飞书、企微、Telegram、钉钉等聊天工具



这意味着你不需要打开新的APP，不需要学新的界面，拿起手机发条消息就能指挥它干活



它满足了一个非常强烈的心理需求：**我的助理随时随地都在**



但从技术角度讲，这就是一个消息适配器。把聊天软件的消息格式转成内部格式，然后统一处理。



这是工程化的事，不是智能化的事



**第二个：7×24 小时运行 + 长期记忆**



7×24小时很简单，你只要把它部署在一台服务器上，只要电脑不关机，它就能一直运行，等待你的指令



而很多人会跟你讲 OpenClaw 可以记住你做过的事情，你不需要每次都重复告诉他。这听起来特别像 AI 觉醒了，有了记忆



但本质是OpenClaw 把你的偏好、你的规则都存了下来，下次用的时候直接进行加载，这是数据库的事，不是 AI 的事。



**第三个：定时任务**



它可以主动推送日报、周报，主动提醒你完成某件事



从被动接受指令到主动通知，这个方向的转变，让很多人第一次觉得自己真的拥有了一个在跑着的助理



但本质是一个定时任务系统。每天晚上 12 点，系统自动触发一个任务，把"拉数据、写内容、做封面"这一套流程跑一遍，然后把结果推给你



## 3）OpenClaw火的原因



所以你现在明白它为什么火了：**不是因为AI更聪明了，而是因为工程化的封装，让AI第一次住进了普通人的工作流里**



但这里有一个很重要的推论，你一定要记住：**工程化的东西，可以被复刻**



OpenClaw是开源的，任何人都可以看它的代码



事实上现在已经有KimiClaw、MaxClaw、元气Bot等相继出现，国内的大模型厂商已经开始跟进



这类工程化能力会越来越快地被固化成标准产品，门槛只会越来越低



这意味着**会装OpenClaw，不是你的竞争力**



我们后面的整篇文章都在给大家讲，什么才是你的竞争力



## 4）**这个框架有什么用？**



这个"智能化 vs 工程化"的框架，不只适用于 OpenClaw



下次你看到任何 AI 产品，你都可以问自己：这个产品里，哪些是智能化的突破，哪些是工程化的包装？



如果一个AI产品的智能化没有突破，那它的价值就主要来自工程化。而工程化的东西，往往很快就会被复制



所以，你要判断的是：这个工程化的方案，到底有多大的价值？它能解决什么问题？它的边界在哪里？



这样，你就不会被自媒体的叙事带偏，你能看清楚，一个产品到底值不值得学、以及应该学什么



# 三、OpenClaw的工程架构拆解：四个主角



为什么我要给大家讲 OpenClaw 的工程架构？你可能会说，大圣，我是小白，我不想了解这个，我只想用



先跟你说一下，我作为一个资深程序员，安装OpenClaw的经历



过去的半个月里，我有三次安装OpenClaw的经历。



第一次是使用腾讯云，花了10个小时把OpenClaw配置好了



第二次是在我自己的苹果电脑上，花了3个小时也把OpenClaw配置好了



第三次就是飞书的妙搭推出的一键安装（1分钟）



网上有很多的教程，但没有一个可以手把手教你。如果你连OpenClaw的最基本架构都不了解，那你在配置所谓的飞书，安装所谓的技能的时候，一定会遇到很多坑，而且你不知道怎么解决



所以这一章节非常重要。我不会给你讲技术性的语言，但是我一定要给你建立一个整体的框架



你可以把 OpenClaw 记成一个非常好背的结构：



**入口 → 大脑 → 手脚 → 技能包**



- **入口（Gateway / Channels）**：接收消息，做翻译和分发
- **大脑（Agent Runner）**：理解意图，规划步骤，决定调用哪些工具
- **手脚（执行器 / 沙箱）**：真正去操作文件、命令行、浏览器
- **技能包（Skills）**：让能力可以像插件一样扩展



这套结构的关键价值在于：**把思考和动手分开**



接下来让我用一个具体的例子来串起来



我日常会根据链接提取公众号的文章，这就是我的一个工作流，这个流程我把它装在了 OpenClaw 上，让我们通过一个例子来看下OpenClaw的完整工作流程

首先，我的小龙虾接入了飞书的 OpenClaw，我在飞书里给他下达了一个命令

这里的逻辑就是，这条消息会被发送到一个叫做网关的地方（Gateway）



而网关会把消息送给 OpenClaw 的大大脑，对这个任务进行处理和规划



## 2）大脑：**ReAct 循环**



OpenClaw 的大脑，也就是大模型，接收到消息后会开始做决策



他做的事情通常包括如下几步：



**第一步：加载上下文**



它会加载近期对话（短期记忆）和用户偏好（长期记忆）



**第二步：让模型出计划**



它把用户目标拆成可执行步骤。比如：



1. 根据用户的任务查看有无可以使用的技能
2. 调用技能处理用户的任务
3. 将处理后的结果返回给用户



**第三步：逐步执行**



每一步执行完，把结果喂回模型，再决定下一步。



比如第一步找到了提取公众号文章的技能，模型看到这个结果，决定调用这个技能去处理用户的任务



**第四步：直到结束**



完成、失败、或触发循环次数/时间上限



这里有一个非常关键的认知：**模型只负责决定做什么，真正怎么做由执行器和技能包提供**



## 3）手脚：安全执行



当模型做出规划之后，它会使用手脚执行一些动作

假设我们通过了权限的校验，执行器会开始执行，在这个案例里面，当公众号文章被提取出来以后，我让它帮我写了一个文件

而大模型本身肯定是没有办法写文件的，所以它利用的是电脑的一些操作指令



执行器这一层解决的是：**AI 从只会说变成能动手，它能直接读取、删除你电脑上的文件，也能创建新的文件**



这样它就真正可以成为你的PC端的个人助理

## 4）**技能包：可扩展的能力**



前面三个部分，我们都不用太过感知，这些东西全部都是模型帮我们去做的。



那真正有价值的在哪里？就在技能包上，也就是我们经常说的 Skill



一个技能包通常包含：



- **SKILL.md**：说明书，写清楚这技能能干什么、怎么用、输入输出是什么
- **scripts/**：可执行脚本（Python/Shell/JS），这是手
- **references/**：参考材料（业务规则、API 文档），按需加载



![](https://feishu.cn/file/Ab9ubAjUWoSn7UxZeCFc2VSDnbc)



**Skills 的本质，就是把Workflow 变成了可复用的插件**



![](https://feishu.cn/file/LyiZbOe57oKygjxRFrLc5YafnAV)



OpenClaw 的核心代码大概就 4000 来行。但它的 Skills 仓库里有上万个技能包



就像 App Store 的逻辑：iOS 系统本身保持稳定，真正的生态价值来自上面的 App。



OpenClaw 也是一样：核心框架保持稳定，真正的价值来自社区贡献的 Skills



你装什么能力，就获得什么能力。这个设计非常聪明



## 5）一个关键的认知



现在我相信你已经知道了，技能包的质量决定了 OpenClaw 能不能真的帮到你



Skills 是 OpenClaw 真正的核心。但 Skill 的本质不是技术，是业务



你的 SOP 有多清晰，你的 Skill 就能有多强。你的 Skill 积累有多深，你的 Agent 就能有多好用



工具会变，Skills 不会



---



那技能包从哪里来？有人会说，网上不是有很多现成的技能包吗？直接装就好了



我不反对你用现成的技能包，但我要告诉你一个现实：**市面上95%的技能包，对你的具体业务来说都是鸡肋**



你的业务有你自己的逻辑。你的客户跟进方式、你的内容生产流程、你的每日工作节奏，这些东西是你独有的，没有人能提前帮你写好对应的技能包



你想要的不是技能包，是人家的业务认知



你觉得真正核心的业务认知会有人在互联网上公开出来吗？会有人99就卖给你吗？



**所以记住一句话：真正有价值的技能包，只能是你自己打磨出来的**



那怎么打磨呢？



网上很多人会跟你讲用 OpenClaw 自己去更新迭代技能，这没有问题，但它真的不适合新手



因为它封装得太好了，你在OpenClaw里下一个指令，它去跑，跑完给你一个结果。



但中间发生了什么？它怎么理解你的意图的？它在哪一步出了偏差？为什么这次跑得好下次跑得不好？



你看不见，也感受不到



# 四、另外一个工具：Claude Code / CodeX



Claude Code是Anthropic推出的通用智能体。我更愿意叫它通用智能体，而不是"AI编程工具"，因为它能做的事情远不止编程



它的工作方式很直接：你建一个文件夹，把你想让它处理的材料、背景信息、参考文档都放进去，然后用自然语言告诉它你想做什么。它会在这个文件夹里工作，调工具、写文件、处理任务，最终把结果输出给你



**最关键的一点：它是透明的**



你能看见它每一步在干什么，它读了哪些文件，它调用了什么工具，它在哪里卡住了。



这个透明度，是你打磨技能包的关键条件：你看得见过程，才能发现问题，才能迭代改进



用Claude Code打磨技能包，没有捷径，就是不断地用、不断地改、不断地验证。



但每一个打磨成熟的技能包，都是真正属于你的资产



<callout emoji="✨">
有一个我自己的判断标准：**如果你能用Claude Code把OpenClaw自己装起来，你才算真正把Claude Code玩明白了。** 这个过程本身，就是在训练你用AI解决真实问题的能力
</callout>

# 五、龙虾到底该怎么养？



我们言归正传，上面说了那么多，并不是不让大家去养龙虾，而是想让大家正确地去养



我先说终局：



<callout emoji="✨">
我们所有的目的都是沉淀出自己的**AI工作流**，并且把这个工作流封装成Skill，能够让龙虾自动调用
</callout>



怎样才能沉淀出自己的AI工作流？



这里我要搬出我一直在用的一个公式：懂AI + 懂业务= 用好AI



所以我把养龙虾的顺序分成四步，而且每一步都有它不可跳过的理由



## 1）第一步：搞懂AI的能力边界



在你碰任何AI工具之前，你得先知道AI擅长什么、不擅长什么



它为什么有时候说得头头是道，有时候又一本正经地胡说八道？



你为什么同样一个问题，换一种问法结果差很多？



这些问题想不清楚，你用任何工具都会碰壁，而且不知道问题出在哪里



这一步你只需要花费5个小时，但它是地基。地基不打好，后面全是空中楼阁



所以我们的课程第一阶段叫做：《AI大模型通识》，所有小白学员必须从这里开始



## 2）第二步：重度使用大模型，用到有手感



我还经常说一句话，AI是学不会的，AI是用会的



你只需要花费5个小时去了解了AI的通识，但在未来的3\~5年里面，你需要不断的跟AI协作，不断使用各种AI模型、各种AI工具，从而建立你的AI手感



就好像我们需要和一个人磨合好：我们不仅要提前了解他的脾气性格，这会让我们少走弯路；但同时我们也需要跟他有大量的项目合作，从而加深彼此的了解



---



大模型有很多，国内的 DeepSeek、豆包、千问，国外的 GPT、Claude 和 Gemini



很多人喜欢问这些模型之间有什么区别。但没有人能够告诉你



使用AI就像跟人交朋友，没有一个绝对的准则能够帮你提前判断它适不适合你。只有你真正相处之后，才能得到答案

# 六、养龙虾的5个注意事项



## 1）权限问题



OpenClaw 之所以能干那么多活，



是因为你给了它足够的权限。它可以对这台电脑上所有的文件进行增删改查



对于个人而言，不要在你的主力电脑上去玩龙虾，容易把你的系统折腾坏



对于公司而言，如果你真的要用在生产上，更要注意权限和数据安全的问题



## 2）服务器部署问题



OpenClaw 如果要7×24小时干活，你必须给它一台电脑，而且这个电脑要7×24小时开机



市面上常见的就三种方案：



第一种方案：额外买一台电脑，专门用来部署 OpenClaw。比如Mac Mini或者Macbook Air



从长远来看，性价比最高



比如可以去京东买一台8G + 256G的MacBook Air M1，跑OpenClaw绰绰有余，价格在2500-3000之间



缺点就是一次性投入成本太高，电脑容易吃灰，7 \* 24时开机



---



第二种方案：你在腾讯云、阿里云等云服务厂商自己购买一台服务器来部署 OpenClaw



按月付费，初始成本会很便宜，可能只需要100块左右，并且你不需要永久开着你的电脑



但本地的主力电脑和云端的服务器电脑之间的数据通信、文件迁移，这些会难住很多小白



---



第三种方案：大模型厂商直接提供一键解决方案，你什么都不用配，只要每个月付钱，就有一个国产版的 OpenClaw 给你用



它只能用模型厂商本身的API



而你的OpenClaw想达到完全体，你就得给它配最强的海外模型，但模型厂商不会给你这样一个灵活度



比如 Kimi 的 KimiClaw、Minimax 的 MinimaxClaw，以及 Coze 也能够一键部署

然后等待简单易用的替代 OpenClaw、或者说更方便安装的方案出

## 3）消耗Token



龙虾比对话机器人消耗token，这是它的技术方案决定的



有人会说，我们用相对便宜的模型，50块钱可以跑一整天



但这里有个前提，你的工作流已经打磨得非常成熟，已经明确知晓便宜模型可以稳定运行  
  
但现实是，在前期打磨工作流的过程中，你根本不知道哪个模型最适合。这时候一般会出现两种情况。  
  
你用便宜的模型，会发现结果总是不满足预期。哪怕50块钱可以跑一整天，但结果不符合预期，一点用都没有



用顶级的模型，随便聊几句，100块钱就没了



所以龙虾消耗token跟顶级模型和便宜模型无关。你如果真的想把它养好，就做好付钱的准备



这也很好理解，我们经常说数字员工，你见过谁家的好员工一个月50块钱就可以打发的？



大几千



## 4）网络和不稳定的问题



网络和不稳定其实是两码事，但它们经常是关联在一起的。



比如我们因为解决不了网络问题，所以要用国内的中转去使用海外顶级的模型。但这个时候中转API经常不稳定  
  
半夜你要跑一个任务，打算明天早上要用，结果半夜的时候这个API不稳定导致任务中断了，总不能半夜起来检查



再比如现在伊朗和美国冲突，结果把 Anthropic 的机房给炸了。那整个 Claude 的 API、官网的 API 还有中转的 API 都变得非常不稳定

# 七、学OpenClaw到底在学什么

**我们并不是在学 OpenClaw，我们学的是AI时代与智能体沟通协作的新范式**



如果你只是花 99 学 OpenClaw 的安装和操作，除了浪费你的时间和精力之外，没有一点价值



我总结3条不会过时的原则

**第二，积累高质量的Skills才是真正的资产：**不管工具怎么变，你沉淀的业务SOP、Skills都是跟着你走的。换一个模型，换一个工具，你可以十分钟内就全部适配过去



**第三，亲自上手，用出手感**：如果你一定要玩 OpenClaw，那你要具备耐心，像管理一家公司那样，一步一步去调教它，舍得花时间、舍得花精力、舍得花钱
